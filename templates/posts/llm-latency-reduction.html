How I Reduced LLM Latency by 98%

Published: Feb 2026 • 5 min read

Introduction When building my multi-agent travel planning system, LLM latency was the single biggest bottleneck. Average response time was ~12–18 seconds — unacceptable for production. Through architectural redesign, caching, async orchestration, and dynamic model routing, I reduced latency by 98%, bringing responses down to sub-second in common cases. This article breaks down exactly how.

The Initial Problem

The architecture looked like this: Client → FastAPI → LLM → External APIs → LLM → Response Problems: Every request hit the LLM No caching Blocking API calls Large model used for all queries Redundant reasoning steps Average latency breakdown: LLM inference: 8–12s External APIs: 3–5s Aggregation + formatting: 1–2s Total: ~15 seconds.

Step 1: Redis Caching (Biggest Win)

Tech stack:

FastAPI

Redis

Strategy

Cache:

Normalized user queries

Flight API responses

Final LLM outputs (for repeated patterns)

Example logic:

Hash normalized input

Check Redis

If hit → return immediately

If miss → compute + store

Impact

Repeated queries dropped from 15s → 150ms.

That alone reduced average latency by ~80%.

Step 2: Async External API Calls

I replaced blocking API calls with async concurrency:

asyncio.gather(
    fetch_flights(),
    fetch_weather(),
    fetch_hotels()
)

Instead of sequential 3–5 seconds:
Parallel execution → ~1.5 seconds.

Latency improvement: ~60% on fresh queries.

Step 3: Dynamic Model Routing

Originally, I used one large model for everything.

Instead:

Small model → simple queries

Larger model → complex reasoning

Deterministic path → no LLM

Routing logic:

Token count

Intent classification

Required reasoning depth

This reduced:

Inference cost by ~60%

Latency by ~40%

Step 4: Eliminating Redundant Reasoning Loops

I initially used multi-step reasoning for every query.

Optimization:

Only invoke reasoning chain when required

Skip planning step for structured requests

This removed 2–3 seconds per request.

Step 5: Smart API Response Storage

External data (flight results) from:
Amadeus IT Group

Instead of:
LLM interpreting raw API JSON each time

I:

Structured responses once

Stored normalized output

Passed compact summaries to LLM

Token reduction → lower inference time.

Key Lessons

Most latency is architectural, not model-related. Caching beats model optimization. Parallelism is mandatory. Token reduction directly reduces cost + time. Production LLM systems require systems engineering, not prompt tweaks.
