How I Reduced LLM Latency by 98%

Published: Feb 2026 • 5 min read

Introduction

When building my multi-agent travel planning system, LLM latency was the single biggest bottleneck. Average response time was ~12–18 seconds — unacceptable for production.

Through architectural redesign, caching, async orchestration, and dynamic model routing, I reduced latency by 98%, bringing responses down to sub-second in common cases.

This article breaks down exactly how.

The Initial Problem

The architecture originally looked like this:

Client → FastAPI → LLM → External APIs → LLM → Response

Main problems:

  • Every request hit the LLM
  • No caching layer
  • Blocking API calls
  • Large model used for all queries
  • Redundant reasoning steps

Average latency breakdown:

  • LLM inference: 8–12s
  • External APIs: 3–5s
  • Aggregation + formatting: 1–2s

Total latency: ~15 seconds.

Step 1: Redis Caching (Biggest Win)

Tech stack:

  • FastAPI
  • Redis

Strategy:

  • Cache normalized user queries
  • Cache flight API responses
  • Cache final LLM outputs for repeated queries

Example logic:

hash = hash_query(user_input)
cached = redis.get(hash)

if cached:
    return cached
else:
    result = compute_response()
    redis.set(hash, result)
    return result

Repeated queries dropped from 15s → 150ms. This alone reduced average latency by roughly 80%.

Step 2: Async External API Calls

I replaced blocking API calls with async concurrency. 

asyncio.gather(
    fetch_flights(),
    fetch_weather(),
    fetch_hotels()
)

Instead of sequential 3–5 seconds of API latency, parallel execution reduced it to roughly ~1.5 seconds.

Latency improvement: about 60% on fresh queries.

Step 3: Dynamic Model Routing

Initially, I used a single large LLM for every request.

Instead, I introduced routing logic:

  • Small model → simple queries
  • Large model → complex reasoning
  • Deterministic path → no LLM required

Routing signals included:

  • Token count
  • Intent classification
  • Required reasoning depth

This reduced inference cost by ~60% and latency by ~40%.

Step 4: Eliminating Redundant Reasoning Loops

The original system used multi-step reasoning chains for every query.

Optimization strategy:

  • Invoke reasoning chains only when required
  • Skip planning steps for structured requests

This removed roughly 2–3 seconds from many requests.

Step 5: Smart API Response Storage

Flight data was retrieved from external travel APIs. Instead of sending raw JSON to the LLM every time, I:

  • Structured responses once
  • Stored normalized outputs
  • Passed compact summaries to the LLM

Reduced token usage significantly and lowered inference time.

Final Results

Stage Average Latency
Initial System ~15 seconds
After Caching ~3 seconds
Async APIs ~1.5 seconds
Model Routing ~1 second
Final Optimized System <300ms (cache hits)

Key Lessons

  • Most latency problems are architectural, not model related.
  • Caching often provides larger gains than model optimization.
  • Parallel API execution is essential.
  • Reducing token usage lowers both latency and cost.
  • Production LLM systems require strong systems engineering, not just prompt design.
