Mitigating Hallucinations Using External API Validation

Published: Feb 2026 • 5 min read

The Problem

Large Language Models can fabricate structured information when they are uncertain. This phenomenon is commonly referred to as hallucination.

In a travel planning system, hallucinating critical information such as:

  • Flight numbers
  • Ticket prices
  • Departure schedules

is unacceptable because users may rely on that information to make real-world decisions.

In early testing, the LLM occasionally generated plausible but completely fabricated flight details when the prompt lacked sufficient context.

The Solution: External API Validation

Instead of allowing the LLM to generate structured travel data, the system retrieves verified information from an external aviation API.

Flight data is sourced from external travel infrastructure providers such as Amadeus.

The LLM is responsible only for interpreting user intent and formatting responses, not for generating factual flight data.

System Architecture

User Query
      ↓
LLM Intent Extraction
      ↓
External Flight API Call
      ↓
Schema Validation Layer
      ↓
LLM Response Formatting
      ↓
User Response

This architecture ensures that all structured data returned to the user originates from a verified external source rather than the model itself.

Validation Pipeline

The validation process works in several stages:

  1. LLM extracts structured query fields from the user request.
  2. System calls the external flight API.
  3. Returned data is validated against a strict schema.
  4. The LLM formats the validated data into natural language.

Example Extracted Query

{
  "origin": "NYC",
  "destination": "LAX",
  "date": "2026-05-10"
}

Guardrails Implemented

Several safeguards were introduced to eliminate hallucinations:

  • JSON schema enforcement for all structured outputs
  • Strict field validation on API responses
  • Response integrity checks before passing to the LLM
  • No free-text generation allowed for flight details

If validation fails at any stage, the system returns a controlled error message rather than allowing the model to fabricate a response.

Results

Metric Before Validation After Validation
Hallucinated Flight Data Occasional 0 occurrences
System Reliability Moderate High
User Trust Risk of misinformation Verified responses

By enforcing API validation, hallucinated flight details were completely eliminated.

Core Insight

LLMs should be used for:

  • Understanding user intent
  • Natural language generation
  • Summarizing structured data

They should not be trusted to invent critical structured information.

Production-grade AI systems must combine LLM reasoning with deterministic validation layers and authoritative data sources.
